# Carcara Rebuild — Implementation Plan

This folder breaks the rebuild of the Carcara Grasshopper plugin into a sequence
of small, self-contained subplans. Each plan is meant to be executed end-to-end
in a single working session: read the file, do the work, run the listed checks,
commit, then move to the next file.

The plan is built around one principle: **prove the loop end-to-end with a
trivial component first, then layer real functionality on top**. Concretely:

1. Get the repository skeleton clean (Phase 0).
2. Get the `.py → .ghuser` build pipeline working (Phase 1).
3. Ship one pointless-but-real component end-to-end and click it on the
   Grasshopper canvas (Phase 2 — the **smoke test**).
4. Only after that loop is green, start porting real components subsystem by
   subsystem (Phases 3–12).

This avoids the worst failure mode: writing 30 components and then discovering
the build pipeline is broken or the GH installation path is wrong.

## Conventions used in every plan

- All paths are repo-root relative. No `carcara-rebuild/` wrapper.
- The deployable folder is `release/`. The package lives at
  `release/crc_modules/` and its **import name is `crc_modules`**.
- Domain logic always lands in `release/crc_modules/<submodule>/` first, then
  the GH bundle in `build/components/CRC_<Name>/`.
- Each GH component is the **componentizer bundle layout**:
  `build/components/CRC_<Name>/{metadata.json, code.py, icon.png}`.
  Flat — subcategory is a `metadata.json` field, not a folder.
- Every plan that adds Python to `crc_modules/` adds tests in `tests/` with
  `psycopg2` mocked. No live DB in unit tests.
- **Coordinate correction (projected coords).** Any component that reads/writes
  geometry across the DB boundary takes `Cx`/`Cy` text inputs (false origin) and
  applies them **in SQL only** (`ST_Translate`) — read subtracts, write/filter adds.
  `Cx`/`Cy` are numeric **text**, **never `float()`-parsed** (that would reintroduce
  the precision loss the system exists to prevent). Helpers in `utils/correction.py`.
  WKT⇄GH converters stay correction-free. See `CLAUDE.md → Coordinate Correction`.
- The build artifact is `release/userobjects/CRC_<Name>.ghuser`, generated by
  `tools/build_userobjects.py` and **committed** (shipped to users).

## Feeding me the component metadata

Several phases need the exact inputs/outputs/behavior of a legacy `.ghuser`.
For each one I'll list, in the **Inputs you must give me** section, the
legacy filename. Open it in Grasshopper (drag onto the canvas after copying
to `%APPDATA%\Grasshopper\UserObjects\`) and copy back:

- Component **name** and **nickname**.
- Each **input parameter**: name, type hint, item/list/tree access, default.
- Each **output parameter**: name and short description.
- A 2-3 sentence description of what the component does at runtime.
- Anything weird (UI menus, dropdown enums, hidden behavior).

That is enough for me to write both `metadata.json` and `code.py` correctly.

## How to install for testing

Components need BOTH the `.ghuser` files AND the `crc_modules` package on
Rhino's path. `tools/deploy.ps1` does both in one shot by mirroring the whole
`release/` folder into `%APPDATA%\Grasshopper\UserObjects\carcara\`.

Every "Grasshopper checkpoint" section assumes this install step:

**Windows**
```powershell
python tools/build_userobjects.py
powershell -ExecutionPolicy Bypass -File tools\deploy.ps1
```

Then **restart Grasshopper** (close + reopen Rhino, or reload GH) — `.ghuser`
files are scanned at startup. The component appears under the **Carcara** tab,
in the subcategory declared in `metadata.json`.

Copying only a `.ghuser` (without the folder) gives
`No module named 'crc_modules'` — always deploy the whole `release/` folder.
If a component disappears after a code change, rebuild + redeploy.
