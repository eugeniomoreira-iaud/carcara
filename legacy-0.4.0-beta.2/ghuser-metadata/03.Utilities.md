# 03.Utilities

Helpers used alongside 02.Queries: the connection-string builder, the generic ODBC
query/command runners, the SQL composer, geometry⇄WKT conversion, SRID lookup, and the
coordinate-correction false origin. Subcategory exposure ordering: `1` primary → `3` tertiary.

## Reference Code

### RunODBCQuery

**Source:** [`scripts/RunODBCQuery.py`](scripts/RunODBCQuery.py)

**Logic:** Guards on `CToggle`, validates `CString` and `Query`, then delegates to `carcara_odbc.run_query_to_tree()` and `carcara_odbc.get_query_headers()` to return two parallel DataTrees — `QResult` with result columns as branches and `QHeaders` with column names mirroring that structure. Errors surface via `out` and GH runtime messages; the component name `runQueryTree.py` and subcategory `03.Utilities` are set at runtime.

---

### RunODBCCommand

**Source:** [`scripts/RunODBCCommand.py`](scripts/RunODBCCommand.py)

**Logic:** Guards on `CToggle`, validates inputs, then delegates to `carcara_odbc.run_command()` for DDL/DML execution. Parses the feedback string for `"success: true"` / `"error"` / `"success: false"` markers and reports rows-affected when present. Returns the raw feedback as `Fb`; all errors go to `out` and GH runtime messages.

---

### GrasshopperGeometryToWKT

**Source:** [`scripts/GrasshopperGeometryToWKT.py`](scripts/GrasshopperGeometryToWKT.py)

**Logic:** Accepts GH geometry as an item, list, or DataTree. Resolves GUID references to actual `Rhino.Geometry` objects, then classifies each as `point`, `linestring`, or `polygon` (closed polylines/curves = polygon; open = linestring). Enforces uniform type across the entire input — mixed types raise an error. Single items produce standard WKT (`POINT`, `LINESTRING`, `POLYGON`); multiple items in a branch produce `MULTI*` variants. DataTree structures are preserved branch-by-branch. Correction is never applied — geometry is assumed to be already in local coordinates.

---

### WKTtoGrasshopperGeometry

**Source:** [`scripts/WKTtoGrasshopperGeometry.py`](scripts/WKTtoGrasshopperGeometry.py)

**Logic:** Iterates over a list of WKT strings, validates each for a recognized geometry-type prefix, and converts via `carcara_geometry.construct_gh_geom()`. Real multipart geometries (where the result is a list with more than one item) are placed together in a single DataTree branch; single or forced-MULTI* items each occupy their own branch. Failed conversions produce empty branches and raise GH warnings. Correction is never applied.

### ConnectionString

**Source:** [`scripts/ConnectionString.py`](scripts/ConnectionString.py)

**Inputs confirmed:** `CToggle` (bool), `DB` (str — database name only; host/port/user/password collected via dialog)
**Outputs confirmed:** `out` (str), `CString` (str)

**Password encoding scheme:** The component opens an Eto modal dialog (`SampleEtoConnectionDialog`) to collect the server IP, username, and password. The password is encoded with `base64.b64encode(password.encode("utf-8")).decode("utf-8")` — standard Base64, **not** encryption. The script's own security notice states: *"Base64 encoding is obfuscation, NOT encryption."* The encoded password is then embedded directly in the ODBC conninfo string as `Pwd=<base64_string>;`.

**CString format:** An ODBC conninfo string with the following structure:
```
Driver={PostgreSQL Unicode(x64)};Server=<ip>;Port=5432;Database=<db>;Uid=<user>;Pwd=<base64_password>;CommandTimeout=0;Timeout=0
```

**Logic:** Guards on `CToggle` and non-empty `DB`, then shows the Eto dialog. On confirmation, encodes the password with Base64 and assembles the ODBC connection string via `_make_connection_string()`. Returns the string as `CString`. No connection test is performed at this stage — reachability is verified downstream.

> **Discrepancy vs old TODO:** The old stub said "confirm the CString format and password-encoding scheme." The decoded script confirms: Base64 (`base64.b64encode`), ODBC conninfo format (not libpq). The new rebuild must switch to a libpq conninfo format (`host=... port=... dbname=... user=... password=...`) to work with `psycopg2` — see `CLAUDE.md` → Connection model.

### SRID

**Source:** [`scripts/SRID_interface.txt`](scripts/SRID_interface.txt)

**Logic:** A native GH ValueList component with no embedded script. Presents a dropdown of 10 common SRID codes for UTM zone 24S: SAD69 / UTM zone 24S (29184, 29194), SIRGAS 2000 / UTM zone 24S (31984), SIRGAS 1995 / UTM zone 24S (31999), WGS 72 / UTM zone 24S (32324), WGS 72BE / UTM zone 24S (32524), WGS 84 / UTM zone 24S (32724), Aratu / UTM zone 24S (20824), Corrego Alegre / UTM zone 24S (22524). The selected integer EPSG code flows directly into downstream components as `SRID`.

### FindCorrectionParameters

**Source:** [`scripts/FindCorrectionParameters_interface.txt`](scripts/FindCorrectionParameters_interface.txt)

**Logic:** A native GH cluster (no top-level Python script). Internally wires SQLComposer and RunODBCQuery to query a single row of a specified table by matching a `Column` against a `Value`, extracting the coordinate columns (`Correction X`, `Correction Y`) from that row. The result is split via native GH text-split and replace-text components into separate `Cx` and `Cy` text outputs. Cx/Cy are returned as strings (text), never converted to float — consistent with the coordinate-correction contract in `CLAUDE.md`.

### SQLComposer

**Source:** [`scripts/SQLComposer.py`](scripts/SQLComposer.py)

**Placeholder syntax — confirmed:** Free-form substring replacement. The `var` list contains the exact substring to find in the template; the `val` list contains the replacement. There are **no delimiters** (no `#`, `{}`, `*`, or similar required wrappers) — the engine calls `statement.replace(placeholder_str, replacement_str)` for each pair. Callers are responsible for choosing placeholder strings that are unique in the template and for ensuring values are safe to embed (no escaping is applied). Mismatched list lengths raise an error; unmatched placeholders log a warning but do not error.

> **Note:** The old 02.Queries.md pseudocode used several different placeholder conventions in the same file — `SCHEMA`, `TABLE`, `*SCHEMA*`, `'FILTER'`, etc. — all of which work because the replacement is a plain string `.replace()`. Callers may use any convention as long as the `var` substring is unique in the template.
